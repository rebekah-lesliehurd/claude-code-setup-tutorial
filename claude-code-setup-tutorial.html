The Terminal
Git Bash — Your Terminal on Windows

Opening the Terminal

The terminal is already on your Mac — you just need to find it.

1

Press ⌘ Command and Space bar at the same time. A search bar appears — this is Spotlight.

2

Type Terminal and press Return.

3

A window opens with a blinking cursor. That's the terminal.

💡 Tip

Keep Terminal in your Dock: right-click its icon and choose Options → Keep in Dock.

What You're Looking At

You'll see something like this:

Terminal prompt

This is the prompt — the terminal's way of saying "I'm ready." The % at the end is where you start typing.

Why Git Bash?

Windows has its own terminals (Command Prompt and PowerShell), but Claude Code requires a Unix-style terminal to run. Git Bash provides this, and it comes bundled with Git for Windows — so you get both tools in one install.

Installing Git for Windows (includes Git Bash)

1

Go to git-scm.com/downloads/win and click Download for Windows.

2

Run the installer. Most defaults are fine. On the screen "Choosing the default editor", change from Vim to Notepad (or Visual Studio Code if you have it) — Vim is very unfamiliar to beginners.

3

On "Adjusting your PATH environment", leave the default: Git from the command line and also from 3rd-party software.

4

Accept all other defaults and click Finish.

Opening Git Bash

1

Press the Windows key, type Git Bash, and press Enter.

2

A dark window opens with a blinking cursor. This is Git Bash — your terminal for the rest of this session.

💡 Tip

Pin Git Bash to your taskbar: right-click its icon while it's open and choose Pin to taskbar.

02
Git & GitHub

What is Git?

Git is a tool that tracks the history of changes you make to files. Think of it like Google Docs version history — except it works for any file, and you control when each snapshot is saved. Every snapshot is called a commit.

What is GitHub?

GitHub is a website that stores your Git history in the cloud. Git runs on your computer; GitHub stores your work online. Together they give you:

  • A backup that isn't tied to your computer
  • A way to share your work with others
  • A place to publish websites for free (covered in the main tutorial)

Step 1: Install Git

The easiest way on a Mac is through the Xcode Command Line Tools:

Terminal
xcode-select --install

A pop-up will appear — click Install and wait a few minutes. Then verify:

Terminal
git --version

Any version number means Git is installed.

Step 1: Verify Git is Installed

Git was installed alongside Git Bash in Module 1. Confirm it's working:

Git Bash
git --version

You should see a version number. If not, re-run the Git for Windows installer.

Step 2: Tell Git Who You Are

Run these two commands, replacing the placeholders with your own details:

Terminal — run both lines
Git Bash — run both lines
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

Then confirm the settings were saved:

Terminal
Git Bash
git config --list

Look for lines starting with user.name and user.email in the output.

📝 Note

Use the same email address you'll use for your GitHub account.

Step 3: Create a GitHub Account

1

Go to github.com and click Sign up.

2

Choose a username — it will appear in your project URLs.

3

Verify your email address when the confirmation arrives.

⚠️ Watch out

Email verification can take a few minutes. Check your spam folder if you don't see it.

Step 4: Set Up SSH Authentication

SSH is the recommended way to connect your terminal to GitHub. It's more secure than a password, and once set up you'll never be prompted for credentials again.

SSH works with a key pair: a private key that stays on your computer, and a public key you share with GitHub. When you push code, they match up automatically.

1

Check whether you already have an SSH key:

Terminal
Git Bash
ls ~/.ssh

If you see id_ed25519 and id_ed25519.pub, skip to step 3. Otherwise continue.

2

Generate a new SSH key (use your GitHub email):

Terminal
Git Bash
ssh-keygen -t ed25519 -C "you@example.com"

Press Return to accept the default location. When asked for a passphrase, press Return to skip — macOS Keychain handles security automatically.

Press Enter to accept the default location. When asked for a passphrase, you can press Enter to skip, or enter one for extra security.

3

Add the key to your Mac's SSH agent:

Terminal
ssh-add --apple-use-keychain ~/.ssh/id_ed25519

Start the SSH agent and add your key:

Git Bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
4

Copy your public key to the clipboard:

Terminal
pbcopy < ~/.ssh/id_ed25519.pub
Git Bash
clip < ~/.ssh/id_ed25519.pub
5

Add it to GitHub:

  • Go to github.com/settings/keys
  • Click New SSH key
  • Give it a name like My MacBookMy Windows PC
  • Paste into the Key field (⌘VCtrl+V) and click Add SSH key
6

Test the connection:

Terminal
Git Bash
ssh -T git@github.com

If asked whether to trust github.com, type yes and press ReturnEnter. You should see: Hi username! You've successfully authenticated.

✅ You did it!

The "successfully authenticated" message means Git and GitHub are connected. You won't need to enter credentials again on this machine.

03
Connecting Git and GitHub

Clone a Public Repository

To clone a repository means to download a complete copy — including its full history — to your computer.

1

Navigate to your home folder:

Terminal
cd ~
Git Bash
cd ~
2

Clone the tutorial repository:

Terminal
Git Bash
git clone git@github.com:WomenDefiningAI/claude-code-skills.git
3

Move into the folder that was created:

Terminal
Git Bash
cd claude-code-skills
4

Open the folder to see what was downloaded:

Terminal
open .
Git Bash
explorer .

Take a moment to browse the files. In Module 5 you'll ask Claude to explain what's here.

Create Your Own Private Repository

1

On GitHub, click the + icon (top-right) → New repository.

2

Name it my-first-repo and set the visibility to Private.

3

Leave "Initialize this repository" unchecked — you'll push files from your computer.

4

Click Create repository. Keep this tab open — you'll need the SSH URL shortly.

Push Your First Code to GitHub

1

Create a new project folder in your home directory:

Terminal
Git Bash
cd ~
mkdir my-first-repo
cd my-first-repo
2

Initialize it as a Git repository:

Terminal
Git Bash
git init
3

Create a placeholder file:

Terminal
Git Bash
echo "# My First Repo" > README.md
4

Stage the file, then check what Git is about to commit:

Terminal
Git Bash
git add .
git status

You'll see README.md listed under "Changes to be committed" in green — Git's way of showing what will be included in the next snapshot.

5

Save the snapshot, then view your commit history:

Terminal
Git Bash
git commit -m "First commit"
git log --oneline

You'll see a one-line entry with a short ID and your commit message. Every commit you make will appear here — this is your project's full history.

6

Connect your local folder to GitHub. On the GitHub page you kept open, copy the SSH URL (starts with git@github.com:) and paste it here:

Terminal
Git Bash
git remote add origin git@github.com:YOUR-USERNAME/my-first-repo.git
7

Push your code to GitHub:

Terminal
Git Bash
git push -u origin main
✅ You did it!

Refresh your GitHub repository page — README.md should have appeared. You just pushed code from your computer to the cloud, with no passwords required.

Branches: Exploring Ideas in Parallel

A branch is an independent copy of your project where you can try something new without touching the main version. If the result is good, you merge it back. If not, you discard it. This is especially powerful with Claude Code — Claude works in a branch, and you only merge when you're happy with the result.

Create a branch and switch to it

Terminal
Git Bash
git checkout -b my-first-branch

The -b flag means "create and switch." Run git branch to confirm — the current branch has a * next to it.

Make a change, commit, and push the branch

Terminal
Git Bash
echo "Experimenting on a branch" >> README.md
git add .
git commit -m "Try something on a branch"
git push -u origin my-first-branch

Merge back into main when you're satisfied

Terminal
Git Bash
git checkout main
git merge my-first-branch
git push

Worktrees: Truly Working in Parallel

Branches are great, but switching between them means stopping what you're doing and changing context. Worktrees solve this — they let you have multiple branches checked out in separate folders at the same time.

With Claude Code, this is a game-changer. You can have Claude working on a new feature in one worktree while you review or test something else in another — no interruptions, no context switching.

Create a new worktree with its own branch

Terminal
Git Bash
mkdir -p ../worktrees
git worktree add -b feature-idea ../worktrees/feature-idea

This creates a new folder called worktrees/feature-idea next to your project, already on a new branch called feature-idea. Both folders share the same Git history.

Work in the new worktree

Terminal
Git Bash
cd ../worktrees/feature-idea
# Make changes, run Claude, commit — without touching your main folder

See all your active worktrees

Terminal
Git Bash
git worktree list

Merge and remove the worktree when done

Terminal
Git Bash
cd ~/my-first-repo
git merge feature-idea
git push
git worktree remove ../worktrees/feature-idea
📝 The Claude Code workflow

Before asking Claude to work on any significant change, create a new worktree. Claude works in its own folder, you can review the main project at the same time, and merging is a deliberate choice — not an accident.

Claude Code Installation

Create a Claude Account

Claude Code requires a Claude Pro subscription or higher (Pro, Max, Team, or Enterprise).

1

Go to claude.ai and sign up or log in.

2

If you don't have a paid plan, go to claude.ai/upgrade. The Pro plan ($20/month) is sufficient for this tutorial.

Install Claude Code

Terminal
Git Bash
curl -fsSL https://claude.ai/install.sh | bash

This downloads and installs Claude Code automatically. It may take a minute or two.

Verify the Installation

Terminal — check version
Git Bash — check version
claude --version

You should see a version number. Then run the health check:

Terminal — health check
Git Bash — health check
claude doctor

Look for a message confirming everything is OK. If anything shows as failing, let your facilitator know.

✅ Done signal

claude doctor passing is your confirmation for this module. You're fully installed and ready.

First Steps with Claude Code

Launch Claude Code

Navigate into the tutorial repository you cloned earlier:

Terminal
cd ~/claude-code-skills
Git Bash
cd ~/claude-code-skills

Launch Claude Code:

Terminal
Git Bash
claude

The first time you launch, you'll be prompted to log in with your Claude account — follow the on-screen instructions. This only happens once. Once authenticated you'll see the prompt:

Claude Code prompt
>

Explore the Repository

Instead of starting from scratch, ask Claude to explain what's already in this repository:

Your prompt to Claude
Please explore this repository and give me a summary of how it's organized,
what the main files and folders are, and what this project appears to do.

Claude will read through the files and write you a plain-English summary. When it asks permission to read files, type y and press ReturnEnter.

Ask a Follow-Up

Once Claude responds, go deeper:

Your prompt to Claude
Which file would be the best starting point to understand how this project works?

This shows how Claude Code is conversational — it remembers everything it already explored in this session.

Exiting and Resuming Sessions

When you're done, exit the session:

Claude Code
/exit

Claude Code saves your conversation automatically. Two ways to pick up where you left off:

Resume the most recent session instantly

Terminal
Git Bash
claude --continue

Reopens your last conversation with full context intact.

Choose from past sessions

Inside Claude Code
/resume

Shows a list of past sessions — useful when you've been working across multiple projects or worktrees.

📝 Why this matters

Claude Code sessions can be long and context-rich. Being able to resume means you never lose your place — whether returning after a break or switching between a feature worktree and a bug fix.
